# VCM-OS — что это и что умеет (простыми словами)

**VCM-OS** = **V**alidated **C**ontext **M**emory **O**perating **S**ystem.

Это система памяти для AI-агентов, которая пишет, читает и проверяет память так, чтобы агент не галлюцинировал и не использовал устаревшую информацию.

---

## Аналогия

Представьте, что агент — это программист, который работает над проектом. VCM-OS — это его **блокнот + система контроля версий + фактчекер**:

- Когда агент пишет код → VCM записывает, что изменилось
- Когда агент принимает решение → VCM фиксирует его и почему
- Когда тест падает → VCM запоминает ошибку и как её чинили
- Когда агент отвечает пользователю → VCM проверяет, правда ли это

---

## Что умеет сейчас

### 1. Память разных типов
Не просто "текст", а структурированная память:

| Тип | Что хранит | Пример |
|-----|-----------|--------|
| **Решение (Decision)** | Что решили и почему | "Используем PostgreSQL, потому что ACID" |
| **Ошибка (Error)** | Что сломалось и как чинили | "Memory leak в кеше, заменили LRU на TTL" |
| **Цель (Goal)** | Что хотим достичь | "Рефакторинг auth для v1.2" |
| **Код (Code change)** | Что поменялось в коде | "Добавили JWT middleware" |
| **Инструмент (Tool)** | Результат команды | "pytest: 15 passed, 0 failed" |
| **Факт (Fact)** | Общая информация | "Текущая ветка: feature/auth" |

### 2. Проверка ответов агента (Verifier)
Перед тем как отдать ответ пользователю, VCM проверяет:

- ❌ Не использует ли агент отменённое решение?
- ❌ Не противоречит ли агент сам себе?
- ❌ Не врёт ли про результаты тестов?
- ❌ Не выдумал ли файлы/функции, которых нет?
- ⚠️ Нет ли слишком сильных утверждений без доказательств?
- ⚠️ Не забыл ли агент сослаться на источник?

**Если проверка не прошла** → VCM может:
- Попросить агента переделать ответ
- Дополнить контекст новой информацией
- Пометить проблему для человека

### 3. Автоматические связи между памятью
Когда записывается новая память, VCM автоматически связывает её с похожей:

- Две записи из одной сессии → связь
- Решение и файл, который оно меняет → связь
- Ошибка и исправление → связь
- Похожие ключевые слова → связь

Результат: **85% памяти было "осиротевшей" (без связей), теперь 0%**.

### 4. Защита секретов
Перед сохранением VCM вычищает:
- API ключи
- Пароли
- JWT токены
- AWS ключи
- Приватные ключи
- Строки подключения к БД

Всё заменяется на `[REDACTED]`.

### 5. Поддержка многих языков программирования
VCM умеет индексировать код на:
Python, JavaScript, TypeScript, Rust, Go, Java, C, C++.

Находит функции, классы, интерфейсы — и связывает их с решениями и ошибками.

### 6. Чтение результатов инструментов
VCM понимает вывод:
- Тестов (pytest, jest, cargo test)
- Линтеров (mypy, eslint, clippy)
- Docker, Kubernetes, Terraform
- API-запросов (curl)
- Сканеров безопасности

Из каждого результата извлекаются факты и ошибки.

### 7. Поддержка нескольких проектов
- Каждый проект изолирован
- Можно найти похожие проекты и предупредить: "В похожем проекте было такое решение"
- Можно переносить предупреждения между проектами

### 8. Исправления от человека
Если агент ошибся, человек может:
- Пометить память как устаревшую
- Пометить как неверную
- Закрепить важную память
- Удалить

Все исправления записываются в аудит-лог.

### 9. Дашборд
В реальном времени показывает:
- Сколько памяти, какого типа
- Сколько ошибок за 24 часа
- Средний возраст памяти
- Сколько памяти без связей
- Сколько исправлений сделано

### 10. Авторасширение контекста
Если для ответа недостаточно информации, VCM автоматически:
- Переписывает запрос
- Ищет ещё
- Добавляет недостающее в контекст

| Что | Результат |
|-----|-----------|
| Малая нагрузка (200 memories) | Запрос 55ms, инжест 10 mem/s |
| Средняя нагрузка (10k events → 20k memories) | Запрос 107ms, инжест 4.3 mem/s |
| База данных (20k memories) | ~100 MB |
| Тесты | **116/116 проходят** |
| CI gates | **10/10 проходят** |

## Что НЕ умеет (пока)

1. **Прямая интеграция с живым агентом** — сейчас это runtime, но агент ещё не зовёт его автоматически перед каждым шагом
2. **Картинки и диаграммы** — только текст
3. **Распределённое хранение** — одна SQLite база, не шардированная
4. **Real-time сессии** — оптимизировано для batch, не для потоковой обработки

## Как использовать

```python
from vcm_os.memory.writer import MemoryWriter
from vcm_os.storage.sqlite_store import SQLiteStore
from vcm_os.storage.vector_index import VectorIndex
from vcm_os.storage.sparse_index import SparseIndex

store = SQLiteStore("my_project.db")
writer = MemoryWriter(store, VectorIndex(), SparseIndex())

# Записать событие
writer.capture_event(EventRecord(
    event_type="assistant_response",
    raw_text="Decision: use Redis for caching",
    project_id="proj_1",
    session_id="sess_1",
))

# Получить контекст для ответа
pack = builder.build(project_id="proj_1", query="What cache to use?")
# pack содержит релевантную память в ~70 токенах

# Проверить ответ агента
verifier = ResponseVerifier()
result = verifier.verify(response, pack, memories)
# result["passed"] = True/False
```

## Метрики качества

| Метрика | Значение | Что означает |
|---------|----------|--------------|
| Holdout restore | **1.000** | В 100% synthetic сценариев система находит правильные цели/решения/ошибки |
| Decision recall | **1.000** | Все решения находятся |
| Error recall | **0.650** | 65% ошибок находятся (сложнее, чем решения) |
| Orphan ratio | **0.0%** | Вся память связана |
| Token efficiency | **69.4** | Средний pack — 70 токенов (очень компактно) |
| Secret redaction | **12/12** | Все 12 типов секретов вычищаются |
| Verifier checks | **10** | 10 разных проверок на каждый ответ |

## Архитектура простым языком

```
Пользователь спрашивает → VCM ищет в памяти → Строит compact pack
                                      ↓
                              Если pack недостаточен → Авторасширение
                                      ↓
                              Отдаёт агенту → Агент генерирует ответ
                                      ↓
                              Verifier проверяет → Если плохо → Repair loop
                                      ↓
                              Tool output → Автоингест → Обновление памяти
                                      ↓
                              Auto-linker связывает новое со старым
```

**Хранение:** SQLite (события + память + связи) + векторный индекс (семантический поиск) + sparse индекс (keyword поиск).
